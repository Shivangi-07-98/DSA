Q8 What design decisions did you make and why?

Answer:
Key design decisions I made for the chat system:

1. Socket.IO over Plain WebSockets:
   - Decision: Use Socket.IO library instead of native WebSocket API
   - Why: Socket.IO provides automatic reconnection, fallback mechanisms, and room management out of the box. This reduces complexity and improves reliability.

2. React Context for Global State:
   - Decision: Use Context API instead of Redux for chat state
   - Why: The chat state wasn't complex enough to warrant Redux. Context API is simpler and sufficient for our needs. Redux would add unnecessary boilerplate.

3. Split-Panel Layout:
   - Decision: Sidebar for chat list + main area for active conversation
   - Why: This is a familiar pattern (like Slack, WhatsApp Web) that users understand intuitively. It allows quick switching between conversations.

4. Optimistic UI Updates:
   - Decision: Show messages immediately when sent, before server confirmation
   - Why: Improves perceived performance. Users see their message instantly, making the app feel more responsive.

5. Pagination for Chat History:
   - Decision: Load chat history in chunks (pagination) instead of all at once
   - Why: Prevents initial load time issues with long conversation histories. Improves performance and reduces memory usage.

6. Virtualized Message List:
   - Decision: Use virtualization for rendering long message lists
   - Why: Only render visible messages, improving performance for conversations with hundreds of messages.

7. Tailwind CSS for Styling:
   - Decision: Use Tailwind utility classes instead of CSS modules or styled-components
   - Why: Faster development, consistent design system, easier maintenance. Tailwind's utility-first approach fits well with component-based architecture.

8. Separate REST API for History, Socket.IO for Real-time:
   - Decision: Use REST API to fetch initial chat history, Socket.IO only for new messages
   - Why: REST API is more reliable for initial data load. Socket.IO handles real-time updates efficiently. This hybrid approach gives us the best of both.

These decisions were based on performance, maintainability, and user experience considerations.

Q9 What challenges did you face while building this feature?

Answer:
Several challenges I encountered:

1. Socket.IO Connection Management:
   - Challenge: Handling reconnections, especially when users switch networks or the app goes to background
   - Solution: Implemented reconnection logic with exponential backoff. Added connection status indicators so users know when they're disconnected.

2. Message Ordering and Duplicates:
   - Challenge: Messages could arrive out of order or get duplicated due to network issues or reconnections
   - Solution: Implemented message deduplication using unique message IDs. Added timestamp-based sorting to ensure correct order.

3. Performance with Long Chat Histories:
   - Challenge: Loading and rendering hundreds of messages caused performance issues
   - Solution: Implemented pagination and virtual scrolling. Only load visible messages, load more as user scrolls up.

4. Real-time State Synchronization:
   - Challenge: Keeping UI in sync with real-time updates while maintaining good UX
   - Solution: Used optimistic updates for sent messages, proper state management with Context API, and careful handling of race conditions.

5. Mobile Responsiveness:
   - Challenge: Chat interface needed to work well on small screens
   - Solution: Created responsive layout that switches to full-screen chat on mobile, hiding sidebar. Used Tailwind's responsive utilities.

6. Integration with Backend:
   - Challenge: Coordinating Socket.IO events and REST API calls, ensuring data consistency
   - Solution: Clear communication with backend team, defined event contracts, thorough testing of integration points.

7. Handling Offline Scenarios:
   - Challenge: What happens when user goes offline mid-conversation?
   - Solution: Implemented message queuing. Messages are stored locally when offline, sent when connection is restored. Added visual indicators for message status (sending, sent, failed).

8. Typing Indicators Performance:
   - Challenge: Typing indicators could cause too many Socket.IO events
   - Solution: Implemented debouncing - only send typing events after user stops typing for a certain duration, reducing server load.

These challenges taught me a lot about building robust real-time applications and handling edge cases.

Q10 How did you handle state management? Why that approach?

Answer:
I used a hybrid approach combining React Context API and local component state.

Global State (Context API):

Why Context API:
- The chat state needs to be accessible across multiple components (sidebar, chat window, header)
- Context API is simpler than Redux for our use case - we don't need time-travel debugging or complex middleware
- It's built into React, no additional dependencies
- Sufficient for our state management needs without over-engineering

Local Component State (useState/useReducer):

Why Local State:
- These states are only relevant to specific components
- No need to share across components
- Keeps components self-contained and easier to reason about
- Better performance - avoids unnecessary re-renders

I avoided Redux because:
- Our state isn't complex enough to need Redux
- Redux adds boilerplate code
- Context API is sufficient and more React-idiomatic
- Easier for other developers to understand and maintain

8. Bundle Size:

9. Network Optimization:
   - Implemented request batching where possible
   - Used compression for API responses

10. Rendering Optimization:
    - Avoided inline object/array creation in render

Q13 Can you discuss an example of caching or performance optimization you implemented?

Answer:
One significant optimization I implemented was caching chat history and user profiles.

Problem:
- Every time user opened a chat, we were fetching full chat history from API
- User profiles were fetched repeatedly for the same users
- This caused unnecessary API calls and slower load times
- Poor experience when switching between chats

Solution - Multi-level Caching:

1. In-Memory Cache (React State):
   - Cached chat history in Context state
   - Once loaded, chat history stays in memory
   - When user switches back to a chat, we use cached data
   - Only fetch new messages that arrived while chat was inactive

2. Local Storage Cache:
   - Stored user profile data in localStorage (refresh after 1 hour)
   - Reduced API calls for frequently accessed profiles

3. API Response Caching:
   - Used React Query (or similar) for automatic caching
   - Cached API responses with appropriate TTL
   - Automatic background refetching for stale data
   - Deduplication of simultaneous requests

Results:
- Reduced API calls by ~60% for chat-related requests
- Chat switching became instant (no loading spinner)
- Improved perceived performance significantly
- Reduced server load
- Better offline experience (cached data available)

This caching strategy significantly improved user experience, especially for users who frequently switch between multiple chats.

Q14 What was one incident where something went wrong in production? How did you solve it?

Answer:
Incident: Chat messages were not appearing for some users after a deployment.

Timeline:
- Deployed a new version with performance optimizations
- Within 30 minutes, started receiving reports that messages weren't showing
- Affected about 20% of active users

Initial Investigation:
1. Checked error logs - found Socket.IO connection errors
2. Tested on my end - worked fine for me
3. Realized it was environment-specific

Root Cause:
- In the optimization, I had changed how we handle Socket.IO connection initialization
- The new code had a race condition where connection was established before authentication token was set
- This worked in development but failed in production due to different timing

Solution:
1. Immediate Fix: Rolled back to previous version (took 5 minutes)
   - Used Vercel's rollback feature
   - Service restored for all users

2. Proper Fix:
   - Fixed the race condition by ensuring token is available before connecting
   - Added proper loading states
   - Added error handling for connection failures

3. Prevention:
   - Added integration tests for Socket.IO connection
   - Improved error logging to catch such issues earlier
   - Added monitoring alerts for connection failures

Lessons Learned:
- Always test in staging environment that mirrors production
- Consider timing differences between environments
- Have a quick rollback plan ready
- Better error logging helps in faster debugging
- Don't optimize without thorough testing

The incident was resolved within 30 minutes, and we put measures in place to prevent similar issues.
