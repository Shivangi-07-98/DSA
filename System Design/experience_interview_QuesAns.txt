Q11 What design decisions did you make and why?

Answer:
Key design decisions I made for the chat system:

1. Socket.IO over Plain WebSockets:
   - Decision: Use Socket.IO library instead of native WebSocket API
   - Why: Socket.IO provides automatic reconnection, fallback mechanisms, and room management out of the box. This reduces complexity and improves reliability.

2. React Context for Global State:
   - Decision: Use Context API instead of Redux for chat state
   - Why: The chat state wasn't complex enough to warrant Redux. Context API is simpler and sufficient for our needs. Redux would add unnecessary boilerplate.

3. Split-Panel Layout:
   - Decision: Sidebar for chat list + main area for active conversation
   - Why: This is a familiar pattern (like Slack, WhatsApp Web) that users understand intuitively. It allows quick switching between conversations.

4. Optimistic UI Updates:
   - Decision: Show messages immediately when sent, before server confirmation
   - Why: Improves perceived performance. Users see their message instantly, making the app feel more responsive.

5. Pagination for Chat History:
   - Decision: Load chat history in chunks (pagination) instead of all at once
   - Why: Prevents initial load time issues with long conversation histories. Improves performance and reduces memory usage.

6. Virtualized Message List:
   - Decision: Use virtualization for rendering long message lists
   - Why: Only render visible messages, improving performance for conversations with hundreds of messages.

7. Tailwind CSS for Styling:
   - Decision: Use Tailwind utility classes instead of CSS modules or styled-components
   - Why: Faster development, consistent design system, easier maintenance. Tailwind's utility-first approach fits well with component-based architecture.

8. Separate REST API for History, Socket.IO for Real-time:
   - Decision: Use REST API to fetch initial chat history, Socket.IO only for new messages
   - Why: REST API is more reliable for initial data load. Socket.IO handles real-time updates efficiently. This hybrid approach gives us the best of both.

These decisions were based on performance, maintainability, and user experience considerations.

Q13 How did you handle state management? Why that approach?

Answer:
I used a hybrid approach combining React Context API and local component state.

Global State (Context API):
- Active chat conversations list
- Currently selected chat
- Socket.IO connection status
- Unread message counts
- User authentication data

Why Context API:
- The chat state needs to be accessible across multiple components (sidebar, chat window, header)
- Context API is simpler than Redux for our use case - we don't need time-travel debugging or complex middleware
- It's built into React, no additional dependencies
- Sufficient for our state management needs without over-engineering

Local Component State (useState/useReducer):
- Message input text
- Scroll position in chat window
- UI states (loading, error messages)
- Form field values
- Component-specific UI toggles

Why Local State:
- These states are only relevant to specific components
- No need to share across components
- Keeps components self-contained and easier to reason about
- Better performance - avoids unnecessary re-renders

State Structure Example:
```
Global Context:
- chats: Array of chat objects
- activeChatId: Currently selected chat
- connectionStatus: 'connected' | 'disconnected' | 'connecting'
- unreadCounts: Map of chatId to unread count

Local State (ChatWindow component):
- messages: Array of messages for active chat
- inputText: Current input value
- isTyping: Whether user is typing
- scrollPosition: For maintaining scroll on new messages
```

I avoided Redux because:
- Our state isn't complex enough to need Redux
- Redux adds boilerplate code
- Context API is sufficient and more React-idiomatic
- Easier for other developers to understand and maintain

This approach provides a good balance between simplicity and functionality.


Q14 How did you optimize performance?

Answer:
Several performance optimizations I implemented:

1. Code Splitting:
   - Used Next.js dynamic imports for chat components
   - Lazy loaded the chat feature so it doesn't block initial page load
   - Split vendor bundles to reduce initial bundle size

2. Virtual Scrolling:
   - Implemented virtual scrolling for message list
   - Only render messages visible in viewport
   - Significantly improved performance for long conversations (1000+ messages)

3. Memoization:
   - Used React.memo for message bubble components to prevent unnecessary re-renders
   - Memoized expensive computations using useMemo
   - Used useCallback for event handlers passed as props

4. Pagination:
   - Load chat history in chunks (20-30 messages at a time)
   - Load more as user scrolls up
   - Reduced initial load time from seconds to milliseconds

5. Debouncing:
   - Debounced typing indicators (only send after 500ms of no typing)
   - Debounced search in chat list
   - Reduced unnecessary Socket.IO events and API calls

6. Image Optimization:
   - Lazy loaded images in chat (avatars, attachments)
   - Used Next.js Image component for automatic optimization
   - Implemented progressive loading for better UX

7. State Optimization:
   - Avoided unnecessary state updates
   - Used functional updates to prevent stale closures
   - Minimized Context value changes to reduce re-renders

8. Bundle Size:
   - Tree-shaking unused code
   - Used smaller alternatives where possible (Day.js instead of Moment.js)
   - Analyzed bundle size regularly using webpack-bundle-analyzer

9. Network Optimization:
   - Implemented request batching where possible
   - Used compression for API responses
   - Cached static data (user profiles, clinic info)

10. Rendering Optimization:
    - Avoided inline object/array creation in render
    - Used keys properly in lists
    - Prevented unnecessary component mounts/unmounts

These optimizations resulted in:
- Initial load time reduced by ~40%
- Smooth scrolling even with 500+ messages
- Reduced memory usage
- Better user experience overall

Q17 Can you discuss an example of caching or performance optimization you implemented?

Answer:
One significant optimization I implemented was caching chat history and user profiles.

Problem:
- Every time user opened a chat, we were fetching full chat history from API
- User profiles were fetched repeatedly for the same users
- This caused unnecessary API calls and slower load times
- Poor experience when switching between chats

Solution - Multi-level Caching:

1. In-Memory Cache (React State):
   - Cached chat history in Context state
   - Once loaded, chat history stays in memory
   - When user switches back to a chat, we use cached data
   - Only fetch new messages that arrived while chat was inactive

2. Local Storage Cache:
   - Stored user profile data in localStorage
   - Key: `user_profile_${userId}`
   - TTL: 1 hour (refresh after 1 hour)
   - Reduced API calls for frequently accessed profiles

3. API Response Caching:
   - Used React Query (or similar) for automatic caching
   - Cached API responses with appropriate TTL
   - Automatic background refetching for stale data
   - Deduplication of simultaneous requests

Implementation Details:

```typescript
// User Profile Cache
const getUserProfile = async (userId: string) => {
  const cacheKey = `user_profile_${userId}`;
  const cached = localStorage.getItem(cacheKey);
  
  if (cached) {
    const { data, timestamp } = JSON.parse(cached);
    const age = Date.now() - timestamp;
    
    if (age < 3600000) { // 1 hour
      return data;
    }
  }
  
  const profile = await api.getUserProfile(userId);
  localStorage.setItem(cacheKey, JSON.stringify({
    data: profile,
    timestamp: Date.now()
  }));
  
  return profile;
};
```

Results:
- Reduced API calls by ~60% for chat-related requests
- Chat switching became instant (no loading spinner)
- Improved perceived performance significantly
- Reduced server load
- Better offline experience (cached data available)

This caching strategy significantly improved user experience, especially for users who frequently switch between multiple chats.

   - Example: Form, FormField, FormButton working together

7. Observer Pattern:
   - Socket.IO uses observer pattern for event handling
   - Components subscribe to events and react to changes

8. Factory Pattern:
   - Used for creating API client instances
   - Centralized configuration and creation logic

9. Strategy Pattern:
   - Used for different validation strategies in forms
   - Different validation rules based on form type
   - Easy to add new validation strategies

10. Module Pattern:
    - Organized code into modules (features, utilities, components)
    - Clear module boundaries and exports
    - Prevents global namespace pollution

- Feature-based folder structure (not MVC or layered)
- Unidirectional data flow (React's one-way data binding)

Q31 Explain a situation where you had to debug a critical bug.

Answer:
Critical Bug: Messages disappearing in chat after reconnection

Situation:
- Users reported that after reconnecting to the internet, their recent messages disappeared
- This was critical because it affected patient-doctor communication
- Occurred for about 15% of users

Timeline:
- Bug reported on Monday morning
- High priority due to impact on user communication
- Needed quick resolution

Debugging Process:

1. Initial Investigation:
   - Couldn't reproduce immediately on my machine
   - Checked error logs - found Socket.IO reconnection events
   - Reviewed recent code changes related to chat

2. Reproducing the Issue:
   - Simulated network disconnection (Chrome DevTools Network throttling)
   - Disconnected internet, sent messages, reconnected
   - Successfully reproduced: messages sent while offline disappeared

3. Root Cause Analysis:
   - Traced through the code flow:
     - Messages sent while offline were stored in local state
     - On reconnection, Socket.IO reconnected
     - But the reconnection logic was clearing local message state
     - Messages weren't being persisted to backend before disconnection
     - On reconnect, we fetched chat history, but offline messages weren't included

   The Bug:
   ```typescript
   // Problematic code
   useEffect(() => {
     socket.on('reconnect', () => {
       setMessages([]); // Clearing messages on reconnect!
       fetchChatHistory(); // But offline messages weren't in history
     });
   }, []);
   ```

4. Understanding the Flow:
   - When offline: Messages stored only in React state
   - On reconnection: State cleared, history fetched
   - Offline messages never reached backend, so they weren't in history
   - Result: Messages lost

5. Solution:

   Immediate Fix:
   - Stopped clearing messages on reconnect
   - Instead, merge fetched history with local messages
   - Deduplicate based on message ID

   ```typescript
   // Fixed code
   useEffect(() => {
     socket.on('reconnect', async () => {
       const history = await fetchChatHistory();
       
       // Merge local messages with history
       setMessages(prev => {
         const localIds = new Set(prev.map(m => m.id));
         const historyMessages = history.filter(m => !localIds.has(m.id));
         return [...prev, ...historyMessages];
       });
     });
   }, []);
   ```

   Proper Fix:
   - Implemented message queue with persistence
   - Store offline messages in localStorage
   - Retry sending when connection restored
   - Only remove from queue after successful send

   ```typescript
   // Message queue implementation
   const sendMessage = async (text) => {
     const message = createMessage(text);
     
     if (socket.connected) {
       await socket.emit('message', message);
     } else {
       // Store in queue
       const queue = JSON.parse(localStorage.getItem('messageQueue') || '[]');
       queue.push(message);
       localStorage.setItem('messageQueue', JSON.stringify(queue));
     }
   };
   
   // On reconnect, send queued messages
   socket.on('connect', () => {
     const queue = JSON.parse(localStorage.getItem('messageQueue') || '[]');
     queue.forEach(msg => socket.emit('message', msg));
     localStorage.removeItem('messageQueue');
   });
   ```

6. Testing:
   - Tested offline scenario thoroughly
   - Verified messages persist and send after reconnection
   - Tested with multiple messages
   - Tested with app refresh during offline

7. Deployment:
   - Fixed and tested in 4 hours
   - Deployed to staging, verified fix
   - Deployed to production same day
   - Monitored for any issues

8. Prevention:
   - Added integration tests for offline scenarios
   - Improved error handling and logging
   - Added user feedback for message status (sending, sent, failed)

Result:

Q33 How do you prioritize tasks when multiple things are assigned?

Answer:
Prioritization approach:

1. Understanding Priority Levels:

   Critical/Urgent:
   - Production bugs affecting users
   - Security issues
   - Blockers for other team members
   - Time-sensitive deliverables

   Important/High Priority:
   - Feature development with deadlines
   - Performance issues
   - User-reported bugs
   - Technical debt that's blocking progress

   Medium Priority:
   - New features without strict deadlines
   - Improvements and optimizations
   - Code refactoring
   - Documentation

   Low Priority:
   - Nice-to-have features
   - Minor UI improvements
   - Future enhancements

2. My Prioritization Framework:

   Step 1: Gather All Tasks:
   - List all assigned tasks
   - Understand requirements and deadlines
   - Identify dependencies

   Step 2: Categorize:
   - Urgency: When is it needed?
   - Impact: How many users affected? Business impact?
   - Effort: How long will it take?
   - Dependencies: What blocks what?

   Step 3: Rank:
   - Critical + High Impact = Do first
   - High Impact + Low Effort = Quick wins
   - High Impact + High Effort = Plan and schedule
   - Low Impact = Lower priority

3. Communication:

   With Manager/Team Lead:
   - If overloaded, communicate early
   - Discuss priorities and get alignment
   - Negotiate deadlines if needed
   - Get help if something is blocking

   Example Conversation:
   "I have 5 tasks assigned. Based on deadlines and impact, I'm planning to work on:
   1. Critical bug fix (today)
   2. Feature A (this week)
   3. Feature B (next week)
   Does this align with your expectations?"

4. Practical Example:

   Scenario: Multiple tasks assigned on Monday

   Tasks:
   - Critical: Production bug (users can't login) - 2 hours
   - High: Chat feature enhancement - 1 day
   - Medium: Form optimization - 4 hours
   - Low: Documentation update - 1 hour

   My Approach:
   1. Fix critical bug first (immediate impact)
   2. Work on chat feature (high priority, has deadline)
   3. Form optimization (can be done in parallel or after)
   4. Documentation (when I have time)

5. Time Management:

   - Break large tasks into smaller chunks
   - Use time blocking for focused work
   - Avoid context switching too frequently
   - Batch similar tasks together

6. Handling Overload:

   - Communicate capacity honestly
   - Suggest reprioritization if needed
   - Ask for help when stuck
   - Focus on quality over quantity

7. Balancing Act:

   - Don't ignore low-priority tasks completely
   - Sometimes quick wins boost morale
   - Technical debt needs attention too
   - Balance urgent vs important

8. Tools I Use:

   - Task management: Jira, Linear, or simple lists
   - Calendar for time blocking
   - Notes for task details
   - Regular reviews and adjustments

Key Principles:
- User impact first
- Communicate early and often
- Be realistic about capacity
- Quality over speed
- Learn to say no when necessary

This approach helps me manage multiple tasks effectively while maintaining quality and meeting deadlines.
